# Product Hardening Audit

Date: 2026-07-10

This audit reflects the current React + FastAPI implementation in `C:\Users\charl\OneDrive\Desktop\Routix`. Older documents still mention Streamlit, but the deployed frontend is the Vite React application in `frontend/src`.

| Current Page | Current Primary Data Source | Current Backend Read Model | Current API Calls | Current Main Visualization | Current UX Problem | Current Data Consistency Risk | Proposed Change |
| --- | --- | --- | --- | --- | --- | --- | --- |
| Overview | Shipments, hubs, analytics, providers, interventions | Analytics summary, provider status, hub risk, fleet analysis | `/api/analytics/summary`, `/api/providers/status`, `/api/hubs/risk`, `/api/fleet/analyze`, `/api/interventions` | Network flow canvas, queue, activity feed | Flow looked like a process diagram with dominant arrows | Edge labels did not explain which metric was being encoded | Replace with lane-based Network Operations Flow Canvas with modes |
| Live Operations | Simulation state/events, alerts, visual signals | `simulation_state`, `simulation_events`, operational signals | `/api/simulation/state`, `/api/simulation/next`, `/api/simulation/play`, `/api/simulation/pause`, `/api/simulation/reset` | Network flow plus latest event impact | Runtime felt like a manual script player | UI status could say paused while operator expects playback | Add Start/Pause, controlled autoplay, speed, and follow mode using existing backend processor |
| Package Tracking | Synthetic package rows from shipment list | Shipment table plus derived tracking rows | `/api/shipments` | Tracking map and selected-row twin | Map was schematic and route curve was arbitrary | Package marker position was visually unrelated to route geometry | Use cached Jakarta-Bekasi geometry and progress-based package marker |
| Package Reports | Package journey view, digital twin, interventions, visual signals | `package_journey_view`, `shipment_digital_twin`, operational interventions | `/api/packages/{id}/journey-view`, `/api/packages/{id}/digital-twin`, `/api/interventions`, visual signal endpoints | Journey rail, digital twin, timeline, signal cards | Signal cards could become event spam | Repeated scans could look like unrelated AI demos | Collapse signals into state summaries grouped by signal type |
| Hub Operations | Hub risk, hub events, visual signals | `all_hub_risk`, `analyze_hub_service`, operational signals | `/api/hubs/risk`, `/api/vision/hub-occupancy/{hub}`, `/api/forecast/hub-overflow/{hub}`, `/api/operational-signals` | Hub table, process chips, bars | Process chips did not show magnitude or bottleneck semantics | Composite congestion score could be read as a direct process measurement | Add process-flow pressure visualization and clearer KPI labels |
| Route Planning | Routing jobs from shipments, optimizer candidates | Route candidates generated by optimizer | `/api/routes/optimize` | Schematic route map and candidate table | Route map remained a curved schematic | Candidate geometry was underused in the UI | Draw route candidates from cached/API coordinates and keep methodology secondary |
| Fleet & Vehicles | Fleet analysis, vehicles, maintenance | `analyze_fleet`, maintenance rules | `/api/fleet/analyze`, `/api/maintenance/analyze/{vehicle}` | Table-first vehicle workspace | Strongest interaction model; preserve | Synthetic utilization can be mistaken for telematics | Preserve and keep units explicit |
| Analytics | Analytics summary, hub risk | `analytics_summary`, hub risk analysis | `/api/analytics/summary`, `/api/hubs/risk` | Comparison cards, charts, heatmaps | Needed stronger domain navigation | Mixed operational domains can blur interpretation | Add domain navigator and detail list for latest signals |
| Data, Models & Assumptions | Model registry, providers, training status | Model registry, provider health, data sources | `/api/models`, `/api/providers/status`, `/api/training-data/status`, `/api/data-sources` | Registry table and provider cards | Engine availability/fallback text was too raw | Demo-mode engines may look like trained production models | Show fallback state as registry detail and keep demo labels visible |

## Implementation Notes

- No new AI models were added.
- No page hierarchy was replaced.
- Package Reports and Fleet interactions were preserved.
- Live Operations autoplay still calls the existing FastAPI event processor.
- Visual signals remain operational state sources, not standalone AI pages.
